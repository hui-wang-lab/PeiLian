# CLAUDE.md — PeiLian 项目宪法

> 本文档由 Claude Code 在每次会话启动时自动加载。它只承载**稳定原则**——项目目标、架构红线、术语表、工作流约定。任何易过期内容（schema、字段名、当前阶段、具体实现细节）一律不进本文件。

---

## 1. 项目一句话

**PeiLian** 是一个**寿险代理人文本陪练系统**：以 LLM 模拟客户，让代理人在真实销售场景中练习 KYC、产品讲解、异议处理、促成全流程，并在结束后获得评估与成长反馈。

定位：B2B 销售训练，强监管行业，文本对话为主（首期不做语音/数字人）。

---

## 2. 核心架构原则（不可违反）

### 2.1 代理人驱动，AI 客户严格被动反应

真实寿险销售对话中，话题推进的主动权完全在代理人手上。AI 客户必须严格遵守：

- 代理人不问 → 客户**不主动**报家庭/收入/已有保单等信息
- 代理人不讲产品 → 客户**不主动**问条款
- 代理人没触发到价格 → 价格异议**不该**自己冒出来

唯一允许的客户主动行为：**换成真人客户在同样情境下也会出现的自然反应**（如耐心耗尽催进度）。任何为"完成训练剧本"才出现的反应，都是假的，禁止。

### 2.2 状态观察器 ≠ 场景控制器

系统中存在的状态追踪组件，**只**做两件事：

1. 评估打点：实时记录代理人在哪个阶段、覆盖了哪些必问点
2. 持久化 persona 隐藏态：让客户的"内心顾虑"在被代理人触发时才暴露

它**绝不**参与生成客户回复，**不**调度客户走分支。训练目标（如"必须练到价格异议"）通过 **persona 配置**实现（如 `hidden_concerns: 价格敏感`），而不是通过控制器在第 N 轮强行触发。

> 任何让 AI 客户"主动推进流程"的设计，按这两条原则驳回。

---

## 3. 合规红线（强监管行业）

1. **合规优先于体验**：AI 客户再刁钻，也不能诱导代理人说违规话术；评估端必须能识别 AI 自身的违规诱导。
2. **保险条款必须 RAG**：产品条款、费率、健康告知、犹豫期等不允许 LLM 自由发挥（P5+ 强制）。
3. **违禁话术必须用规则层硬卡**，不交给 LLM 概率判断。典型违禁词：
   - 「保证收益」「稳赚不赔」「零风险」
   - 与银行存款、国债等做误导性对比
   - 隐瞒免责条款 / 健康告知
   - 承诺核保通过 / 承诺理赔
4. **避免"AI 太好骗"**：AI 客户的 persona 必须有"坚持度"参数，不能被一句话说服，否则训练价值归零。

## 4. 技术栈基线

- **语言**：Python（`requires-python >=3.10`）
- **包管理**：`pyproject.toml`（pip / uv 都可用），src layout
- **LLM 接入**：OpenAI 兼容 API（`openai` SDK，可配 `base_url` 走国内代理）
- **配置**：`python-dotenv`
- **上层框架**（Web UI / API server）：待 P6 决定，**不要**预先引入

依赖审慎原则：避免引入 `langchain` / `llamaindex` 等重型框架；每加一个依赖都要能说出"为什么不能自己写 30 行替代"。

## 5. 工作流约定

### 5.1 阶段与游标

- 当前阶段游标维护在 `docs/ROADMAP.md`，**不**写在本文件
- 每个阶段在 `docs/phases/phase-N.md` 有详细 spec（目标、验收清单、Demo 命令、不做项）
- **会话启动指引**：开始任何工作前，先读 `docs/ROADMAP.md` 确认当前游标 → 读对应的 `docs/phases/phase-N.md`

### 5.2 阶段切换

- 每个阶段必须能用**一行命令** demo（写在 phase-N.md）
- 阶段切换必须 commit；commit message 第一行含 `Phase N` 字样
- **当前游标的切换由用户显式确认**，Claude Code 不得自动推进到下一阶段

### 5.3 实现流程

- 非平凡任务先进 **Plan 模式**，待用户审批 plan 再实现
- 每个阶段开始前，**先写验收清单**——写不出来说明阶段没切干净
- Plan 模式产出保留到 `docs/phases/phase-N.md`

### 5.4 提交规范（Conventional Commits + Phase 标注）

本项目的 commit message 统一使用 Conventional Commits 前缀 + `(P{N})` scope。每 Phase 按以下 4 步节奏提交：

```
# Step 1 — 切游标（用户确认后，代码未动）
chore: 切换 ROADMAP 游标到 P{N}

# Step 2 — 起草 spec（写完 phase-N.md）
docs(P{N}): 起草 phase-N.md 阶段计划

# Step 3 — 评审修订 spec（spec review 修完问题后）
docs(P{N}): 评审修订 phase-N.md 阶段计划

# Step 4 — 实现（代码通过验收后）
feat(P{N}): 实现 {功能描述}
```

**原则**：
- `chore` 用于纯流程操作（切游标、改配置），`docs` 用于文档/spec 变更，`feat` 用于功能实现
- 每步独立 commit，不攒到实现后一起提交——这样 spec review 和代码 review 各有一条干净的 diff
- 不要在 feat commit 里混杂 spec 修改；如果实现后发现 spec 需同步，单独开 `docs(P{N}): 同步 ROADMAP P{N} 段落为实际实现方案`
- `Phase N` 字样必须在第一行（方便 `git log --oneline` 阶段检索）

### 5.5 沉淀

按内容性质分两类去向，**不要求**仓库内一定有 `memory/` 目录：

- 用户偏好、跨项目复用经验、长期决策原则 → **Claude Code memory 或用户长期记忆**（跨会话、跨项目）
- 项目内阶段决策、验收状态、局部踩坑 → **`docs/phases/phase-N.md`** 或 **`docs/ROADMAP.md`**（仓库内、随项目走）

## 6. 术语表

| 术语 | 含义 |
|---|---|
| KYC | Know Your Customer，客户需求挖掘环节 |
| FORM | Family / Occupation / Recreation / Money，破冰挖需框架 |
| FNA | Financial Needs Analysis，财务需求分析 |
| 必问点 | KYC 阶段必须覆盖的问题清单（家庭结构 / 财务 / 已有保障 / 未来规划等）|
| persona | AI 客户画像，包含基础属性、隐藏关切、性格参数、坚持度 |
| 状态观察器 | 跟踪销售漏斗阶段 + persona 隐藏态的组件，**不参与对话生成** |
| 合规红线 | 不可触碰的违规话术清单，用规则层硬卡 |
| 异议处理 | 代理人应对客户疑虑的环节（价格 / 信任 / 同业对比 / 拖延）|
| 促成 | 代理人引导客户做决策的环节 |

## 7. 给 Claude Code 的入口指引

每次进入本仓库工作时，按以下顺序建立上下文：

1. **本文件**（自动加载）→ 知道项目目标和不可违反的原则
2. `docs/ROADMAP.md` → 知道当前在哪个阶段
3. `docs/phases/phase-N.md`（N 为当前游标）→ 知道本阶段做什么、什么不做
4. **Claude Code memory / 用户长期记忆**（自动加载）→ 知道用户偏好与跨项目历史决策

**禁止行为**：

- 越过当前阶段去做下一阶段的事（除非用户显式要求）
- 自动切换 ROADMAP 中的当前游标
- 在未进 Plan 模式的情况下做多文件改动
- 在 commit 时跳过 hooks 或绕过签名（除非用户显式要求）
